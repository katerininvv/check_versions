# Техническое задание

## Сервис отслеживания обновлений контейнеров с уведомлениями в Telegram и веб-дашбордом

**Версия документа:** 0.2 (решения согласованы)
**Дата:** 19.06.2026
**Статус:** согласовано, готово к реализации

---

## 1. Назначение и цели

Разработать самостоятельно размещаемое (self-hosted) приложение в виде Docker-контейнера, которое:

1. Отслеживает выход новых версий (обновлений) для заданного списка проектов, заданных как ссылки на образы контейнеров (например `ghcr.io/wg-easy/wg-easy:15`).
2. По расписанию (по умолчанию — **каждое воскресенье в 12:00 по московскому времени, MSK / UTC+3**) формирует и отправляет сообщение в Telegram-канал:
   - если есть новые обновления — список с описанием каждого, **переведённым на русский язык**;
   - если обновлений нет — отдельное сообщение об отсутствии обновлений.
3. Предоставляет защищённую веб-страницу (доступную снаружи) с авторизацией по логину и паролю, на которой пользователь может:
   - в разделе настроек задавать ссылки на собственные ресурсы (например `https://150.251.155.157:54321/vitt/`) с логином и паролем к ним; учётные данные хранятся **в зашифрованном виде**;
   - на главной странице видеть плитки/ссылки на эти ресурсы с описанием; по клику открывается уже авторизованная страница ресурса;
   - в отдельном разделе управлять списком отслеживаемых проектов и параметрами слежения.

### 1.1. Цели

- Не пропускать важные обновления используемых сервисов.
- Получать понятную сводку на русском в удобное время.
- Иметь единую панель быстрого доступа к своим сервисам без ручного ввода паролей.

### 1.2. Вне рамок (Out of scope) — на этом этапе

- Автоматическое применение обновлений (только уведомление, не апдейт самих контейнеров).
- Многопользовательский режим с ролями (по умолчанию — один владелец/администратор).
- Мобильное приложение.

## 4. Общая архитектура

Приложение поставляется как один Docker-образ (или docker-compose из нескольких сервисов). Логические модули:

1. **Web/API-сервер** — бэкенд + отдача веб-интерфейса, аутентификация, REST/JSON API.
2. **Модуль слежения за обновлениями (Update Tracker)** — периодически опрашивает реестры и репозитории.
3. **Модуль перевода (Translator)** — переводит описания релизов на русский.
4. **Модуль уведомлений (Notifier)** — формирует и отправляет сообщения в Telegram.
5. **Планировщик (Scheduler)** — запускает рассылку по расписанию (воскресенье 12:00 MSK).
6. **Хранилище данных** — БД для конфигурации, проектов, ресурсов, истории; защищённое хранилище секретов.
7. **Прокси-/авто-вход модуль (Reverse Proxy / Auto-login)** — обеспечивает открытие ресурсов авторизованными.

```
                 ┌─────────────────────────────────────────────┐
   Браузер  ───► │  Web/API сервер (HTTPS, авторизация)         │
                 │   ├─ Дашборд (плитки ресурсов)               │
                 │   ├─ Настройки ресурсов (+ авто-вход)        │
                 │   └─ Управление проектами / расписанием      │
                 └───────────┬─────────────────────┬───────────┘
                             │                     │
              ┌──────────────▼───────┐   ┌─────────▼────────────┐
              │  Update Tracker      │   │  Reverse Proxy /     │
              │  (опрос реестров)    │   │  Auto-login          │
              └──────┬───────────────┘   └─────────┬────────────┘
                     │                             │
        ┌────────────▼─────┐               внешние ресурсы пользователя
        │ Translator (RU)  │
        └────────┬─────────┘
                 │
        ┌────────▼─────────┐      ┌────────────────────┐
        │ Notifier (TG)    │◄─────┤ Scheduler (cron)   │
        └────────┬─────────┘      │ Вс 12:00 MSK       │
                 │                └────────────────────┘
          Telegram-канал

        ┌──────────────────────────────────────────────┐
        │ Хранилище: БД + зашифрованные секреты         │
        └──────────────────────────────────────────────┘
```

## 5. Функциональные требования

### 5.1. Модуль слежения за обновлениями

- **FR-1.** Пользователь может добавить проект, указав ссылку на образ контейнера в формате `[registry/]namespace/image[:tag]`, например:
  - `ghcr.io/wg-easy/wg-easy:15`
  - `docker.io/library/nginx`
  - `quay.io/...`, `registry.gitlab.com/...`, частный OCI-реестр.
- **FR-2.** **Целевой реестр на текущем этапе — Docker Hub** (`docker.io`). Архитектура опроса остаётся расширяемой под GHCR/Quay/GitLab/частные OCI-реестры на будущее, но реализуется и тестируется именно Docker Hub.
- **FR-3.** **Основная стратегия отслеживания — обновление продуктивной версии `:latest`.** Система периодически запрашивает у Docker Hub манифест тега `:latest` и сравнивает его **digest** с ранее зафиксированным. Смена digest у `:latest` трактуется как выход новой продуктивной версии и является сигналом для уведомления.
  - Дополнительно (best-effort) система пытается определить, какому семантическому тегу версии соответствует новый digest (сопоставление digest → version tag), чтобы в сообщении показать «было `v14` → стало `v15`».
  - Иные стратегии (реакция на конкретный semver-уровень, отслеживание фиксированного тега) остаются доступны в модели данных как опциональные, но по умолчанию для проекта используется отслеживание `:latest`.
- **FR-4.** Для каждого проекта можно задать:
  - человекочитаемое имя и описание;
  - фильтр тегов (regex / семантическое правило, например «только стабильные, без `-rc`, `-beta`»);
  - включить/выключить отслеживание;
  - индивидуальное переопределение расписания (опционально).
- **FR-5.** Система определяет «новизну» относительно последнего зафиксированного состояния (last seen) каждого проекта и хранит историю обнаруженных версий.
- **FR-6.** При обнаружении обновления система пытается получить **release notes / changelog** из доступных источников (GitHub/GitLab Releases API, метки образа OCI `org.opencontainers.image.*`, файл CHANGELOG). Если описание недоступно — фиксируется факт обновления без описания.
- **FR-7.** Частота опроса реестров настраивается отдельно от расписания рассылки (например, опрос раз в час, рассылка — раз в неделю), чтобы к моменту отправки данные были актуальны.

### 5.2. Модуль перевода

- **FR-8.** Описания релизов (release notes) переводятся на русский язык перед включением в сообщение.
- **FR-9.** **Провайдер перевода по умолчанию — DeepL API** (внешний API, тариф API Free: 500 000 символов/мес — кратно превышает потребности еженедельного дайджеста; DeepL даёт лучшее качество RU-перевода среди массовых провайдеров). Модуль перевода построен как **подключаемый (pluggable)** — провайдер выбирается в настройках, на будущее поддерживаются альтернативы (Google Cloud Translation, Yandex, Azure Translator). Ключ API задаётся в настройках/переменных окружения и хранится зашифрованно.
- **FR-9a.** Так как описания на Docker Hub часто скудные, для получения **понятного описания «что изменилось»** система best-effort подтягивает release notes из связанного источника (репозиторий проекта на GitHub/GitLab — определяется по ссылкам/меткам образа), затем переводит их через DeepL. При отсутствии внятных notes отправляется факт обновления с доступной информацией (версии, дата, ссылка).
- **FR-10.** Оригинальный текст сохраняется; в сообщении при необходимости можно показать «перевод + ссылку на оригинал».
- **FR-11.** При недоступности переводчика сообщение отправляется с оригинальным текстом и пометкой, что перевод не выполнен (graceful degradation).

### 5.3. Модуль уведомлений (Telegram)

- **FR-12.** Отправка сообщений в **личный чат пользователя** через Telegram Bot API. **Бот уже создан, токен есть.** Для доставки в личный чат требуется `chat_id` пользователя (получается после того, как пользователь напишет боту `/start`; приложение помогает определить `chat_id` — например, через кнопку «Определить chat_id» в настройках). Токен и `chat_id` задаются в настройках и хранятся зашифрованно.
- **FR-13.** Формат сообщения при наличии обновлений (на русском):
  - заголовок с датой;
  - по каждому проекту: имя, было → стало (версии), краткое описание изменений (переведённое), ссылка на проект/релиз.
- **FR-14.** Формат при отсутствии обновлений: краткое сообщение «За прошедшую неделю новых обновлений не обнаружено» с датой/перечнем проверенных проектов.
- **FR-15.** Учитываются ограничения Telegram (длина сообщения ~4096 символов) — длинные сводки разбиваются на несколько сообщений.
- **FR-16.** Кнопка/возможность «Отправить тестовое сообщение сейчас» из веб-интерфейса для проверки настроек.
- **FR-17.** Логирование результата отправки (успех/ошибка) с возможностью повторной попытки.

### 5.4. Планировщик

- **FR-18.** Рассылка по умолчанию: **каждое воскресенье в 12:00 MSK**.
- **FR-19.** Расписание настраивается из веб-интерфейса (день недели, время, часовой пояс). Часовой пояс по умолчанию — `Europe/Moscow`; корректно учитывается при работе контейнера в UTC.
- **FR-20.** Возможность ручного запуска проверки и рассылки «прямо сейчас».
- **FR-21.** Защита от повторной отправки при перезапусках контейнера (idempotency — фиксируется факт отправки за период).

### 5.5. Веб-интерфейс и аутентификация

- **FR-22.** Веб-страница доступна снаружи по HTTPS, защищена логином и паролем.
- **FR-23.** Аутентификация: логин + пароль; пароль хранится в виде криптостойкого хэша (Argon2/bcrypt). Сессии через защищённые cookie (HttpOnly, Secure, SameSite) либо JWT.
- **FR-24.** Меры защиты: ограничение попыток входа (rate limit / временная блокировка), защита от CSRF, по возможности — опциональная двухфакторная аутентификация (TOTP) как улучшение.
- **FR-25.** Первичная инициализация: при первом запуске задаются логин и пароль администратора (через переменные окружения или мастер первичной настройки).
- **FR-26.** Структура интерфейса:
  - **Главная (Дашборд)** — плитки/ссылки ресурсов с описанием; клик → авто-вход (см. §5.6).
  - **Ресурсы** — добавление/редактирование/удаление ресурсов (URL, логин, пароль, описание, иконка, способ авто-входа).
  - **Проекты** — список отслеживаемых образов и параметры слежения (см. §5.1).
  - **Настройки** — Telegram (токен, chat_id), переводчик (провайдер, ключ), расписание, смена пароля.
  - **История/Логи** — журнал обнаруженных обновлений и отправленных уведомлений.

### 5.6. Ресурсы и авто-вход *(ключевой технически сложный пункт — требует согласования)*

> ⚠️ **Важно.** «Открыть уже авторизованную страницу» для произвольного внешнего сервиса — нетривиально, потому что у каждого сервиса своя схема авторизации. Универсальной «волшебной кнопки» не существует. Ниже — реалистичные варианты; для каждого ресурса выбирается подходящий способ.

- **FR-27.** Для каждого ресурса хранятся: URL, описание, иконка/название, **тип авторизации** и учётные данные.
- **FR-28.** Поддерживаемые способы авто-входа (выбираются на ресурс):
  1. **HTTP Basic Auth** — самый надёжный для авто-входа. Учётные данные подставляются автоматически (через встроенный reverse proxy приложения, который добавляет заголовок `Authorization`). Открытие ресурса проксируется через наше приложение под единым доменом.
  2. **Reverse proxy с воспроизведением формы логина** — приложение само выполняет вход (POST формы), получает сессионную cookie и проксирует трафик. Требует индивидуальной настройки полей формы под конкретный сервис.
  3. **Просто ссылка (без авто-входа)** — открывается логин-страница сервиса; пароль можно опционально автоподставить расширением/скриптом на странице (ненадёжно, зависит от сервиса).
  4. **Передача через URL** (если сервис поддерживает токен/ключ в query) — например наш кейс вида `https://host:port/path/?...`.
- **FR-29.** **Согласованный способ авто-входа — встроенный обратный прокси.** Пользователь обращается к ресурсам через наше приложение (единая авторизация на нашей стороне), а приложение прозрачно подставляет сохранённые секреты (заголовок `Authorization` для Basic Auth либо воспроизведение формы логина и проксирование сессии). Это даёт наиболее стабильное «по клику — уже залогинен». Конкретный механизм (Basic Auth / форма) определяется на каждый ресурс по его типу авторизации (см. FR-28).
- **FR-30.** Если ресурс использует self-signed сертификат (как `https://150.251.155.157:54321/...`), предусмотреть опцию «доверять/не проверять сертификат» для этого ресурса (с явным предупреждением).
- **FR-31.** Учётные данные ресурсов **никогда** не отображаются в открытом виде после сохранения (только маскированно), не попадают в логи и не возвращаются по API в расшифрованном виде без необходимости.

## 6. Нефункциональные требования

### 6.1. Безопасность

- **NFR-1.** Все секреты (пароли ресурсов, токены, API-ключи) шифруются в состоянии покоя (at rest) симметричным шифрованием (например AES-256-GCM). Мастер-ключ шифрования задаётся через переменную окружения/секрет и **не хранится** в БД в открытом виде.
- **NFR-2.** Весь внешний трафик — по HTTPS/TLS. **Публикация по IP-адресу без доменного имени, с самоподписанным сертификатом** (сертификат генерируется приложением при первом запуске либо монтируется пользователем). Let's Encrypt не используется (нет домена). Приложение слушает заданный HTTPS-порт; браузер при первом заходе предупредит о самоподписанном сертификате — это ожидаемо.
- **NFR-3.** Пароль администратора — только хэш (Argon2id). Никакого хранения в открытом виде.
- **NFR-4.** Защита от перебора, CSRF, XSS; заголовки безопасности (CSP, HSTS и т.д.).
- **NFR-5.** Принцип минимальных привилегий: контейнер не от root по возможности; секреты монтируются как Docker secrets/env.
- **NFR-6.** Аудит-лог действий администратора (вход, изменение настроек, добавление ресурсов).

### 6.2. Надёжность и эксплуатация

- **NFR-7.** Данные сохраняются в персистентном volume; переживают перезапуск/обновление контейнера.
- **NFR-8.** Healthcheck-эндпоинт; корректное логирование (уровни, ротация).
- **NFR-9.** Идемпотентность рассылки (§FR-21).
- **NFR-10.** Резервное копирование/экспорт конфигурации (опционально, желательно).

### 6.3. Производительность и масштаб

- **NFR-11.** Рассчитано на одного пользователя и десятки–сотни отслеживаемых проектов/ресурсов.
- **NFR-12.** Опрос реестров — с учётом их rate limit (особенно Docker Hub); кэширование результатов.

### 6.4. Совместимость

- **NFR-13.** Развёртывание на Linux x86_64 и ARM64 (multi-arch образ желателен).
- **NFR-14.** Современные браузеры (Chrome, Firefox, Safari, Edge).

## 8. Технологический стек *(зафиксирован)*

- **Бэкенд:** Python + **FastAPI** (быстрая разработка, богатая экосистема: клиенты DeepL/Telegram, планировщик, работа с реестрами; встроенный reverse proxy на `httpx`/Starlette). *Альтернатива на будущее при росте нагрузки прокси — Go.*
- **Планировщик:** APScheduler (cron-триггер с таймзоной `Europe/Moscow`).
- **БД:** **SQLite** (один пользователь — достаточно, простой бэкап одним файлом, персистентный volume).
- **Шифрование секретов:** AES-256-GCM, мастер-ключ из `SECRET_KEY` (env).
- **Фронтенд:** лёгкий интерфейс (минималистичный SPA либо серверный рендеринг) — дашборд, ресурсы, проекты, настройки, история.
- **Опрос Docker Hub:** Registry/Distribution API (манифест и digest тега `:latest`) с учётом rate limit и кэшированием; release notes — best-effort из связанного репозитория.
- **Перевод:** **DeepL API** (pluggable-провайдер).
- **Telegram:** Telegram Bot API (HTTP) → личный чат.
- **Авто-вход:** встроенный обратный прокси (Basic Auth / воспроизведение формы).
- **Упаковка:** Docker (multi-arch x86_64/ARM64), `docker-compose.yml` + `.env`, самоподписанный TLS.

## 9. Развёртывание

- **DEP-1.** Поставка в виде Docker-образа + пример `docker-compose.yml`.
- **DEP-2.** Конфигурация через переменные окружения и/или файл конфигурации:
  - `ADMIN_LOGIN`, `ADMIN_PASSWORD` (или мастер первичной настройки);
  - `SECRET_KEY` (мастер-ключ шифрования AES-256-GCM);

- **DEP-3.** Персистентный volume для БД/секретов.
- **DEP-4.** Документация: README с быстрым стартом, описанием переменных, примером настройки Telegram-бота и reverse proxy для TLS.

## 10. Критерии приёмки (Acceptance Criteria)

1. Контейнер поднимается одной командой `docker compose up`, веб-интерфейс доступен по HTTPS с логином/паролем.
2. Можно добавить проект `ghcr.io/wg-easy/wg-easy:15`, выбрать стратегию слежения; система корректно фиксирует текущую версию.
3. При появлении новой версии (или принудительной проверке) система формирует сообщение с описанием на русском.
4. В воскресенье в 12:00 MSK сообщение приходит в Telegram-канал; при отсутствии обновлений приходит соответствующее сообщение.
5. Можно добавить ресурс с URL/логином/паролем; секреты хранятся зашифрованно и не отображаются открыто.
6. На дашборде по клику на ресурс открывается авторизованная страница (для поддержанного типа авторизации, минимум Basic Auth / reverse proxy).
7. Тестовая отправка в Telegram работает из интерфейса.
8. Данные сохраняются после перезапуска контейнера.

## 11. Согласованные решения

Все ключевые вопросы закрыты:

| № | Вопрос | Решение |
|---|---|---|
| 1 | Провайдер перевода | **DeepL API** (внешний, тариф API Free 500k симв./мес), модуль pluggable |
| 2 | Способ авто-входа | **Встроенный обратный прокси** (Basic Auth / воспроизведение формы) |
| 3 | Telegram | **Личный чат**, бот уже создан (токен есть); нужен `chat_id` |
| 4 | Реестр | **Docker Hub**, архитектура расширяема под другие |
| 5 | Что считать обновлением | **Смена digest продуктивного тега `:latest`** (новая прод-версия) |
| 6 | Публикация наружу | **По IP без домена, самоподписанный TLS-сертификат** |
| 7 | Пользователи | **Один администратор** |
| 8 | Хранилище | **SQLite** (выбрано по решению «реши сам») |
| 9 | Глубина описания | **Понятное описание «что изменилось»**, перевод на русский |

### Оставшиеся уточнения перед сборкой (не блокирующие)
- Нужно будет узнать `chat_id` твоего личного чата (определим через бота на этапе настройки).
- Подтвердить порт публикации HTTPS наружу (по умолчанию возьмём настраиваемый, например `8443`).
